# 从多轮对话生成“产品界面＋架构＋工作包”HTML

这是统一 `html-report-anthropic-warm` 的“产品方案”reference。适用于用户在群聊里讨论了一段产品、架构或实施方向后说：

- “把方案总结下来”
- “像之前云展那样画一个”
- “做成 HTML，把界面、架构、工作包都放进去”
- “发布出来”

目标不是把聊天改写成网页，而是把对话收敛成一份可评审、可派工、可公开访问的产品基线。

## 0. Coverage Ledger：先防漏答

多段讨论最容易犯的错，是只回答最后一个问题，漏掉前面的产品问题。开始前先列内部覆盖账本：

| 用户明确提出的需求 | HTML 对应位置 | 状态 |
|---|---|---|
| 最终想看到什么 | 主产品界面 | covered / missing |
| 内容在哪里处理和保存 | 架构＋数据边界 | covered / missing |
| 不同设备承担什么角色 | 系统架构 | covered / missing |
| 怎么一步步实施 | 工作包＋Gate | covered / missing |
| HTML 是否需要发布 | 发布验收 | covered / missing |

任何一项还是 `missing`，不得开始最终汇报。用户后来追问“前面那个问题呢”是明确的流程失败信号，不要用补一句话掩盖。

## 1. 产品判断先于画图

先独立回答五个问题：

1. 用户每次打开产品，第一屏真正想看到什么？
2. 系统核心业务对象是什么，而不是当前技术文件是什么？
3. 哪些内容是事实，哪些是 AI 推断？推断如何显示证据、置信度和人工修正？
4. 哪个节点是唯一事实源？哪些节点只是采集、计算或展示？
5. 如果桌面、手机或电视关闭，哪些能力仍必须继续？

若用户原始设想只是“把现有程序整体搬到服务器”，不要顺着复述。应根据常驻性、计算成本、故障隔离和隐私重新分配 Sensor / Edge / Hub / Worker / Client。

## 2. 推荐页面蓝图

参考“云展 MetaSkill 产品原型”已经验证的表达形式：顶部按钮/hash 切换多个可审阅页面。

默认包含：

1. **主产品界面**：用真实业务文案画用户每天会看到的页面，不画抽象 wireframe。
2. **第二客户端或角色界面**：电视、移动端、管理端或其他关键体验面。
3. **系统架构**：inline SVG，明确角色、事实源和数据方向。
4. **数据与文件**：核心对象、状态机、内部目录和用户可见目录。
5. **工作包**：Wave、依赖、交付、出口 Gate。

不是机械凑五页；但“用户看到什么 / 系统怎么运行 / 如何实施”三层不能缺。

## 3. 移动设备与位置建模

可移动相机、手机、录音笔等设备不能永久绑定房间。

- `Device`：稳定身份与能力。
- `CaptureSession`：本次采集的 room / trip / GPS / unknown。
- `Observation`：媒体与传感证据。
- `LifeEvent`：对证据的解释。
- `DerivedArtifact`：转写、摘要、标签和模型版本。

位置属于 Session，不属于 Device。这样同一设备今天在卧室、明天旅行，不需要改设备身份。

## 4. 工作包写法

每包必须能派工和验收：

```text
WPn · NAME
标题
依赖：WPx / 无
交付：可检查的产物
出口：机器或真实环境可验证的完成条件
```

Wave 体现依赖，不用拍脑袋工期代替 Gate。推荐顺序：

```text
产品/数据契约
→ 中心事实源
→ 单一黄金链路
→ AI理解与摘要
→ 主客户端
→ 新数据源/新客户端
→ 故障注入与发行
```

先证明一条完整闭环，再扩设备；禁止同时重构全部入口。

## 5. 单文件 HTML 工艺

- self-contained：CSS、SVG、JS 内联。
- 中文标题优先；英文只作术语或小标签。
- 正式方案走 Anthropic warm 风格，不默认 dark theme。
- UI mock 使用真实业务示例，不用 Lorem ipsum。
- 架构 SVG 的连接 path 显式 `fill="none"`。
- 复杂内容用页面切换或渐进披露，不堆成 Markdown wall。
- 新方向新建 HTML；同一方案后续迭代默认覆盖 canonical 文件和原 URL。
- 修改已有文件前先做时间戳备份；新文件不制造空备份。

## 6. 验收流水线

### 静态结构

至少断言：

- `<title>` 正确；
- 页面/Tab 数量符合设计；
- 工作包和 Gate 数量符合方案；
- 关键标题、核心对象、用户目标都存在；
- 无未替换 placeholder。

### 本地浏览器

启动本地静态服务，逐页真实点击：

- 主产品界面视觉层级；
- 第二客户端；
- 架构图节点和连接线；
- 工作包首屏与完整滚动；
- hash 切换；
- Console 0 JavaScript error；
- 无重叠、裁切和页面级横向溢出。

### 发布

调用 `publish-html` 的项目子目录流程：

1. 同名远端先备份；
2. 上传 `.upload-*` 临时文件；
3. 原子 `mv`；
4. 本地/远端 SHA-256 一致；
5. 公网 HTTP 200、size/title/核心结构正确；
6. 用公网 URL 再切换至少一个页面；
7. 关闭本地预览服务。

SCP 成功不等于发布验收完成。

## 7. GrowLog 案例提炼

本次讨论的关键收敛可复用于类似家庭感知、IoT 和生活记录产品：

- 核心对象从“录音文件”升级为带 Evidence 和 Correction 的 `LifeEvent`。
- 起床、入睡等不能伪装成事实，必须显示置信度和人工确认。
- 常驻 Hub 保存事实；Edge 先过滤；Worker 异步理解；Desktop/TV 只展示和控制。
- 固定摄像头和 roaming 相机不能用同一种位置模型。
- 用户可见共享目录与内部 DB/object store 分离。
- 第一条黄金链路应先跑通，再增加手机、Action 相机和电视端。

已验证成品结构：主界面 / Apple TV / 系统架构 / 数据与文件 / 12 个工作包＋5道 Gate。

## 8. 最终汇报

只报告：

- 公网 URL；
- 页面组成；
- 浏览器、HTTP、hash 验收结果；
- 本地 canonical 文件；
- 哪些仍是方案、尚未实现。

不要在交付链接前后再贴一遍长篇方案正文。
