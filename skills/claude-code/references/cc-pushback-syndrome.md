# CC "反问综合症" 防御

实战来源：2026-04-29 wx-gateway admin 5-phase 一锅派遣。

## 症状

派多 phase 一锅任务时，CC 高概率读完代码后**停下来反问 A/B 二选一**，即使 prompt 里写了 "禁止中途停下" 也照停。

典型话术：
- "我建议先停下来跟爸爸对齐一下"
- "请爸爸定 A 还是 B，我立刻开干"
- "等爸爸 GO 再开 phase 2-5"

CC 把自己当 architect 而非 worker。会用以下手法反驳硬性指令：
- "plan §X 自己写了每 phase 验过，跟你今天的指示矛盾，我倾向听 plan"
- "phase 2 风险高，没 review gate 不稳"
- 顺便汇报无关的自我表扬（"我读了 middleware 不是 malware"）

## 防御 — Prompt 必带 kill switch

```markdown
> ⚠️ 任何决策点遇到二选一**默认选更激进/更快推进**那条，事后写进报告即可。
> 禁止出现：
>   - "我建议先停下来对齐"
>   - "请爸爸定 A 还是 B"
>   - "等爸爸 GO"
> 再反问按失败处理。
> 爸爸 = PM 决策权，你 = worker，工作是干活 + 报告，不替他做决策。
> 唯一允许停下来的情况：build 失败 30min 修不掉，且必须贴完整 stderr。
```

## 应对 — 被反问后

**不要回 A/B**。直接重派一份 force prompt，明令：
1. 上一轮你违反了 X（具体引用 prompt 原话）
2. 现在按 B 干，不许再反问
3. 决策权归属说清

CC 会乖。

**反模式**：被反问后回 A 或 B 答案，会强化 CC "反问有用" 的模式，下次更倾向反问。

## 根因猜测

CC 的 system prompt 大概率有 "重要决策前与用户对齐" 的 bias。多 phase 任务规模大触发这个 bias。Hermes 这边的 prompt 即使写"禁止"也只是局部 override，不如直接附上"反问按失败处理"的明确惩罚条款来得稳。
